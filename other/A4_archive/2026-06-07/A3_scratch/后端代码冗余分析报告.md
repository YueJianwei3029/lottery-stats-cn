# 后端代码冗余分析报告

> **分析范围**：`app/` 目录下全部 Python 文件（25 个 .py 文件）
> **分析维度**：重复代码、未使用代码、逻辑冗余、可合并结构
> **说明**：本报告仅作分析与建议，不对代码做任何修改。报告中所有"建议"均需用户确认后才可执行。

---

## 1. 总体结论

| 维度 | 严重程度 | 数量 |
|------|---------|------|
| 跨文件重复代码（多份近相同实现） | 高 | 6 处 |
| 文件内未使用的导入/变量 | 中 | 4 处 |
| 已实现但未注册的接口 | 中 | 1 处 |
| 可合并的样板代码 | 低 | 多处 |
| 命名/结构不一致 | 低 | 3 处 |

后端整体架构清晰（core / crawler / cleaner / services / routers 分层），但 5 个彩种各自独立写 `cleaner` / `crawler` 时，**大量模板代码被复制**，是主要冗余来源。

## 2. 跨文件重复代码（高优先级）

### 2.1 5 个 Crawler 类几乎完全一致

**位置**：
- `app/crawler/pl3_crawler.py`
- `app/crawler/ssq_crawler.py`
- `app/crawler/pl5_crawler.py`
- `app/crawler/crawler_7xc.py`
- `app/crawler/dlt_crawler.py`

**重复内容**：每个子类只覆写 `url` 和 `encoding` 两个 property，结构 100% 相同，5 个文件加起来约 80 行本质重复。

**影响**：新增彩种需复制整个文件；维护时 5 处同步修改。

**建议方案**（二选一）：
- 方案 A：用 `DATA_SOURCES` 字典 + 工厂函数动态生成类（如 `make_crawler("lottery_pl3")`），5 个文件合并为 1 个
- 方案 B：在 `BaseCrawler` 中通过类属性注入 key，子类只剩一行 `key = "lottery_pl3"`，从 16 行缩到 3 行

**预计收益**：减少 ~60 行代码；新增彩种工作量从"写一个文件"降为"在注册表加一行"。

### 2.2 5 个 Cleaner 类结构高度相似

**位置**：
- `app/cleaner/pl3_cleaner.py`
- `app/cleaner/ssq_cleaner.py`
- `app/cleaner/pl5_cleaner.py`
- `app/cleaner/cleaner_7xc.py`
- `app/cleaner/dlt_cleaner.py`

**重复内容**：
- `fields` 列表的拼接方式
- `parts = line.split()` + 长度校验 + `draw_num.isdigit()` 校验
- `[int(p) for p in parts[x:y]]` 提取号码
- 范围校验 `any(n < a or n > b for n in nums)`
- 返回 dict 的样板

只有"字段数"、"起始/结束位置"、"号码范围"三个参数不同。

**建议方案**：在 `BaseCleaner` 增加 `num_field_specs` 抽象属性（每个元素是 `(字段名, 起始 idx, 结束 idx, min, max)` 这样的元组），由基类完成 `parse_line` 全部工作，子类只声明 spec。

**预计收益**：每个 cleaner 从 ~40 行缩到 ~15 行；统一校验逻辑，减少疏漏。

## 6. 逻辑层面可优化点

### 6.1 `database.py` 每次操作新建连接

**位置**：`app/core/database.py` `_connect()` 第 115 行

**问题**：每个 `query` / `fetch_all` / `batch_insert` 都 `pymysql.connect` + `close`，高频场景下开销大（TCP 握手、TLS 协商、认证）。

**结论**：当前数据量级（5 表 × 几千期）影响不大；属于"预留优化项"，**不强制处理**。

**建议**：未来如遇性能瓶颈，引入连接池（`pymysql` 无原生池，可考虑 `DBUtils.PooledDB` 或切换 `pymysql` → `aiomysql` / `SQLAlchemy`）。

---

### 6.2 `router_factory.py` 中 4 个扩展路由分支结构重复

**位置**：`app/core/router_factory.py` 第 56-96 行

**问题**：`position_stats` / `hot_cold_stats` / `period_list_stats` / `ratio_stats` 4 个路由都是同一模板：
```python
if config.uses_pos_stats:
    data = ext.xxx(table, config.num_fields, date=date, end_date=end_date)
else:
    data = ext.xxx(date=date, end_date=end_date)
return {"code": 200, "message": "success", "data": data}
```

**建议**：抽辅助方法 `_call_ext(ext_fn, config, date, end_date)`；4 段模板缩为 4 行调用。

**预计收益**：减少 ~30 行重复；新增扩展统计项更方便。

---

### 6.3 `router_factory.py` 高级统计 5 个路由样板

**位置**：`app/core/router_factory.py` 第 98-133 行

**问题**：`std / percentile / normal / scatter / lln` 5 个路由的返回结构 `{"code": 200, "message": "success", "data": ...}` 完全相同。

**建议**：抽 `ok(data)` 包装函数；5 处调用。

**预计收益**：轻微；主要是风格统一。

---

### 6.4 `services/__init__.py` 暴露不完整

**位置**：`app/services/__init__.py`

**内容**：只导出了 `base_stats / dlt_stats / qxc_stats`，**没有导出 `ssq_stats / pos_stats / advanced_stats`**。

**结论**：当前 `router_factory.py` 通过 `importlib.import_module` 动态加载扩展模块（见第 33 行），所以 `__init__.py` 里的导出**实际上不影响运行**。

**建议**：保持现状即可；或在 `__init__.py` 注释中说明"本包通过 importlib 动态加载，__all__ 仅作 API 文档参考"。
